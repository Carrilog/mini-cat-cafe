# mini-cat-cafe 代码架构分析

## 一、整体架构

mini-cat-cafe 是一个五层架构的多 Agent 框架，各层单向依赖，禁止反向引用：

```
cli/
  └─→ orchestrator/
        └─→ agents/
              └─→ tools/
              └─→ core/
        └─→ core/
providers/
  └─→ core/
```

**依赖规则：** 上层可以引用下层，同层之间不互相引用，providers 只依赖 core。

---

## 二、各层文件与职责

### 1. core/（核心抽象层）

不依赖任何第三方库，只用 Python 标准库。定义全局共享的数据结构和接口契约。

#### `core/types.py`

定义系统内所有数据流转的基础类型：

| 类型 | 字段 | 说明 |
|------|------|------|
| `Role` | USER / ASSISTANT / SYSTEM / TOOL | 消息角色枚举 |
| `Message` | role, content, raw_blocks, metadata | 对话消息，raw_blocks 携带 provider 原始结构化块 |
| `ToolCall` | name, arguments, call_id | LLM 发出的工具调用请求 |
| `LLMResponse` | content, tool_calls, raw_blocks, usage, model, metadata | LLM 返回结果 |

`raw_blocks` 和 `metadata` 是两个关键扩展字段：
- `raw_blocks`：携带 provider 原始协议块（Anthropic 的 tool_use/tool_result，OpenAI 的 tool_calls），在 tool loop 中跨轮传递，确保协议合规
- `metadata`：携带 provider 特有扩展数据（如 DeepSeek R1 的 reasoning_content）

#### `core/llm_interface.py`

```python
class BaseLLMProvider(ABC):
    async def complete(messages, tools, **kwargs) -> LLMResponse: ...
    def get_model_name() -> str: ...
```

所有 provider 必须实现的抽象接口。`complete()` 接收标准 `Message` 列表，返回标准 `LLMResponse`，屏蔽各厂商 API 差异。

#### `core/base_agent.py`

```python
class BaseAgent(ABC):
    def __init__(self, llm: BaseLLMProvider, name: str): ...
    async def run(task, context) -> str: ...  # 子类实现
```

所有 Agent 的抽象基类，持有 LLM 实例和对话历史列表。

### 2. providers/（LLM 适配层）

将各厂商 API 适配为统一的 `BaseLLMProvider` 接口。

#### `providers/anthropic_provider.py`

使用 `anthropic.AsyncAnthropic` 客户端。关键处理：

- **System prompt 分离**：Anthropic API 的 system 参数独立于 messages，provider 在序列化时自动提取 `Role.SYSTEM` 消息
- **Adaptive thinking**：仅对 `4-6` 系列模型启用 `thinking: {"type": "adaptive"}`，其他模型不发送该参数
- **raw_blocks 保留**：将响应的 content 数组（text + tool_use 块）序列化为 raw_blocks，供下一轮 tool loop 回传

```python
# 响应 → raw_blocks（供下轮回传）
raw_blocks = [
    {"type": "text", "text": b.text} if b.type == "text"
    else {"type": "tool_use", "id": b.id, "name": b.name, "input": b.input}
    for b in response.content
]
```

#### `providers/openai_provider.py`

使用 `openai.AsyncOpenAI` 客户端，通过 `base_url` + `env_key` 参数支持 OpenAI、Qwen、Moonshot 等兼容接口。关键处理：

- **Schema 转换**：工具 schema 从 Anthropic 风格（`input_schema`）转为 OpenAI 风格（`parameters`）
- **tool_call_id 绑定**：tool result 消息需携带 `tool_call_id` 与 assistant 的 `tool_calls` 对应
- **角色映射**：`Role.USER + raw_blocks` → 多条 `role: tool` 消息；`Role.ASSISTANT + raw_blocks` → `role: assistant` 带 `tool_calls` 字段

#### `providers/deepseek_provider.py`

DeepSeek 独立 provider，在 OpenAI 兼容基础上处理 DeepSeek 特有协议：

- **reasoning_content 提取**：从响应中提取 `choice.reasoning_content`，存入 `LLMResponse.metadata`
- **reasoning_content round-trip**：在 `_build_api_messages` 中，assistant 消息从 `Message.metadata` 取出 `reasoning_content` 回传给 API（DeepSeek 要求在多轮 tool loop 中必须回传，否则返回 400）
- **prompt cache 统计**：提取 `prompt_cache_hit_tokens` 存入 usage

#### `providers/factory.py`

工厂函数，懒加载各 provider（避免未安装 SDK 时报错）：

```python
def create_provider(provider_name: str, **kwargs) -> BaseLLMProvider:
    if name == "anthropic": ...
    elif name == "openai": ...
    elif name == "deepseek": ...   # 走独立 DeepSeekProvider
    elif name == "qwen": ...       # 复用 OpenAICompatibleProvider
    elif name == "moonshot": ...   # 复用 OpenAICompatibleProvider
```

### 3. tools/（工具层）

每个工具继承 `BaseTool`，实现 4 个成员后即可被任意 Agent 使用。

#### `tools/base_tool.py`

```python
class BaseTool(ABC):
    name: str           # 工具名（LLM 调用时使用）
    description: str    # 工具描述（LLM 决策依据）
    parameters_schema   # JSON Schema，描述参数结构
    async execute(**kwargs) -> str   # 执行逻辑，返回字符串结果

    def to_llm_schema() -> dict:    # 转为 Anthropic 风格 schema（input_schema）
```

`to_llm_schema()` 输出 Anthropic 风格，各 provider 在序列化时自行转换。

#### `tools/web_search.py` — WebSearchTool

使用 `duckduckgo-search` 库，无需 API key。返回标题 + URL + 摘要的 Markdown 格式文本。

#### `tools/file_writer.py` — FileWriterTool

写文件到 `output/` 目录，使用 `Path(filename).name` 防止路径穿越攻击（只取文件名，丢弃目录部分）。

#### `tools/code_runner.py` — CodeRunnerTool

将代码写入临时文件，用 `subprocess.run` 执行，30 秒超时，捕获 stdout + stderr，执行后删除临时文件。

---

### 4. agents/（Agent 实现层）

#### `agents/tool_using_agent.py` — ToolUsingAgent

核心 tool loop 实现，所有具体 Agent 的基类：

```
初始化消息列表 [SYSTEM, USER(task)]
↓
循环（最多 10 次）：
  1. llm.complete(messages, tools) → response
  2. 如果没有 tool_calls → 返回 response.content（结束）
  3. 追加 assistant 消息（保留 raw_blocks + metadata）
  4. 并行执行所有 tool_calls → 收集结果
  5. 将所有结果打包为单条 USER 消息（raw_blocks=tool_result_blocks）
  6. 继续下一轮
↓
超过 10 轮 → 返回超时提示
```

**关键设计**：所有 tool result 合并为一条 USER 消息（Anthropic 协议要求），而非每个结果一条消息。

#### `agents/researcher.py` — ResearcherAgent

工具：`WebSearchTool` + `FileWriterTool`

System prompt 定向为量化金融研究员，要求搜索 → 整理 → 保存 Markdown，并注明来源 URL。

#### `agents/coder.py` — CoderAgent

工具：`CodeRunnerTool` + `FileWriterTool`

System prompt 定向为量化开发者，要求写代码 → 运行验证 → 保存 .py 文件。

---

### 5. orchestrator/（编排层）

#### `orchestrator/orchestrator.py` — Orchestrator

基于关键词的任务路由：

| 关键词 | 路由 |
|--------|------|
| research / search / find / learn / explain / what is | ResearcherAgent |
| code / implement / write / calculate / backtest / strategy / compute | CoderAgent |
| 两者都有 | 先 Researcher，再将结果作为上下文传给 Coder |
| 都没有 | 默认两者都跑 |

**上下文传递**：当 Researcher 先跑时，其输出会拼接到 Coder 的任务描述中，实现跨 Agent 的信息流转。

---

### 6. cli/（入口层）

#### `cli/main.py`

```bash
python cli/main.py --task "..." --provider anthropic --model claude-opus-4-6 --output-dir output
```

参数：
- `--task`：任务描述（必填）
- `--provider`：anthropic / openai / deepseek / qwen / moonshot（默认 anthropic）
- `--model`：模型名，省略则用 provider 默认值
- `--output-dir`：输出目录（默认 output/）

## 三、核心调用流程

### 完整调用链（以 Anthropic + 研究任务为例）

```
用户命令行输入
    ↓
cli/main.py: parse_args()
    ↓
providers/factory.py: create_provider("anthropic") → AnthropicProvider
    ↓
orchestrator/orchestrator.py: Orchestrator.run(task)
    ↓ 关键词匹配 → needs_research=True
agents/researcher.py: ResearcherAgent.run(task)
    ↓ 继承自 ToolUsingAgent
agents/tool_using_agent.py: 构建初始消息 [SYSTEM, USER]
    ↓
providers/anthropic_provider.py: complete(messages, tools=[web_search, write_file])
    ↓ Anthropic API 返回 tool_use 块
ToolUsingAgent: 追加 assistant 消息（raw_blocks=tool_use 块）
    ↓
tools/web_search.py: execute(query="...") → 搜索结果文本
    ↓
ToolUsingAgent: 追加 USER 消息（raw_blocks=[tool_result 块]）
    ↓
providers/anthropic_provider.py: complete(messages) → 最终文本回复
    ↓
tools/file_writer.py: execute(filename="...", content="...") → 保存文件
    ↓
Orchestrator: 返回 {"research": "..."}
    ↓
cli/main.py: 打印结果
```

### Tool Loop 协议细节

Anthropic 和 OpenAI 的 tool calling 协议结构不同，框架通过 `raw_blocks` 字段统一处理：

**Anthropic 协议：**
```
assistant: {content: [{"type": "tool_use", "id": "...", "name": "...", "input": {...}}]}
user:      {content: [{"type": "tool_result", "tool_use_id": "...", "content": "..."}]}
```

**OpenAI 协议：**
```
assistant: {content: null, tool_calls: [{"id": "...", "function": {"name": "...", "arguments": "..."}}]}
tool:      {role: "tool", tool_call_id: "...", content: "..."}
```

`ToolUsingAgent` 使用统一的 `raw_blocks` 结构存储，各 provider 在 `complete()` 内部按自身协议序列化。

## 四、关键技术实现

### 1. 异步并发（asyncio）

整个框架基于 `asyncio`，所有 LLM 调用和工具执行都是 `async/await`。CLI 入口用 `asyncio.run(main())` 启动事件循环。工具执行目前是串行的（for 循环），但接口是 async，后续可改为 `asyncio.gather` 并发执行。

### 2. 工厂模式 + 懒加载

`providers/factory.py` 使用工厂函数，在 `if/elif` 分支内部才 import 对应 provider 模块。好处：未安装某个 SDK（如 `anthropic`）时，使用其他 provider 不会报 ImportError。

### 3. 抽象基类（ABC）

`BaseLLMProvider`、`BaseAgent`、`BaseTool` 均使用 Python `abc.ABC` + `@abstractmethod`，强制子类实现必要接口，在实例化时即报错，而非运行时才发现缺失。

### 4. Python 3.9 兼容性

所有使用 `X | Y` 联合类型注解的文件都加了 `from __future__ import annotations`，使类型注解在 Python 3.9 下作为字符串延迟求值，避免运行时 TypeError。工厂函数用 `if/elif` 替代 Python 3.10+ 的 `match/case`。

### 5. DeepSeek R1 reasoning_content 传递

DeepSeek R1 模型在多轮 tool loop 中要求将上一轮的 `reasoning_content` 回传，否则 API 返回 400。框架通过 `Message.metadata` 字段在轮次间传递：

```
LLMResponse.metadata["reasoning_content"]
    → Message.metadata["reasoning_content"]（ToolUsingAgent 构建 assistant 消息时传入）
    → DeepSeekProvider._build_api_messages() 读取并加入 API 请求
```

## 六、扩展指南

### 添加新 Provider

1. 在 `providers/` 新建 `xxx_provider.py`，继承 `BaseLLMProvider`，实现 `complete()` 和 `get_model_name()`
2. 在 `providers/factory.py` 的 `if/elif` 链中添加新分支
3. 在 `cli/main.py` 的 `choices` 列表中添加新 provider 名

### 添加新工具

1. 在 `tools/` 新建文件，继承 `BaseTool`，实现 `name`、`description`、`parameters_schema`、`execute()`
2. 在需要该工具的 Agent 的 `__init__` 中加入工具实例

### 添加新 Agent

1. 在 `agents/` 新建文件，继承 `ToolUsingAgent`
2. 传入工具列表和 system prompt
3. 在 `orchestrator/orchestrator.py` 中添加路由逻辑
