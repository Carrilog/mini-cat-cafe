# mini-cat-cafe 设计文档

## 项目目标

一个最小可用的多 Agent 量化学习框架，支持：
- 自主从网络搜索信息并整理为文档
- 编写并运行量化相关 Python 代码
- 通过 CLI 接入多家 LLM 厂商

---

## 架构分层

```
mini-cat-cafe/
├── core/           # 核心抽象层（不依赖任何外部库）
├── providers/      # LLM 提供商适配层
├── tools/          # 工具层
├── agents/         # Agent 实现层
├── orchestrator/   # 多 Agent 编排层
├── cli/            # CLI 入口
└── design/         # 设计文档（本目录）
```

**依赖方向（单向，禁止反向依赖）：**

```
cli → orchestrator → agents → tools
                  ↘         ↗
                    core
                  ↗
         providers
```

---

## 各层职责

### core/（核心抽象）

| 文件 | 职责 |
|------|------|
| `types.py` | 数据类型定义：`Message`、`Role`、`ToolCall`、`LLMResponse`（含 `raw_blocks` 和 `metadata` 字段） |
| `llm_interface.py` | `BaseLLMProvider` 抽象接口，所有 provider 必须实现 |
| `base_agent.py` | `BaseAgent` 抽象基类，持有 LLM 实例和对话历史 |

**设计原则：** core 层不 import 任何第三方库，只用 Python 标准库。

### providers/（LLM 适配层）

| 文件 | 职责 |
|------|------|
| `anthropic_provider.py` | Claude 系列，仅 4.6 系列启用 adaptive thinking |
| `openai_provider.py` | OpenAI 兼容接口，复用于 Qwen / Moonshot |
| `deepseek_provider.py` | DeepSeek 独立 provider，支持 deepseek-chat（tool-calling）和 deepseek-reasoner（R1 reasoning_content round-trip） |
| `factory.py` | `create_provider(name, **kwargs)` 工厂函数，懒加载各 provider |

**扩展方式：** 在 `factory.py` 的 `if/elif` 块中添加新分支，无需修改其他文件。

### tools/（工具层）

| 文件 | 工具名 | 功能 |
|------|--------|------|
| `web_search.py` | `web_search` | DuckDuckGo 搜索，无需 API key |
| `file_writer.py` | `write_file` | 写文件到 output/，防路径穿越 |
| `code_runner.py` | `run_python` | subprocess 执行 Python，30s 超时 |

**扩展方式：** 继承 `BaseTool`，实现 4 个抽象成员即可被任意 Agent 使用。

### agents/（Agent 实现层）

| 文件 | Agent | 工具 | 职责 |
|------|-------|------|------|
| `tool_using_agent.py` | `ToolUsingAgent` | 通用 | 工具调用循环基类（最多 10 轮） |
| `researcher.py` | `ResearcherAgent` | web_search + write_file | 搜索 → 整理 → 保存 Markdown |
| `coder.py` | `CoderAgent` | run_python + write_file | 写代码 → 运行验证 → 保存 .py |

**工具调用循环：** `ToolUsingAgent.run()` 实现标准的 LLM → tool_call → tool_result → LLM 循环，子类只需提供工具列表和 system prompt。

### orchestrator/（编排层）

`Orchestrator` 根据任务关键词路由：

| 关键词 | 路由 |
|--------|------|
| research / search / find / learn / explain | ResearcherAgent |
| code / implement / write / calculate / backtest | CoderAgent |
| 两者都有 / 都没有 | 先 Researcher，再将结果作为上下文传给 Coder |

### cli/（入口层）

```bash
python cli/main.py --task "..." --provider anthropic --model claude-opus-4-6 --output-dir output
```

## 关键设计决策

### 决策 1：用 OpenAI 兼容接口统一非 Anthropic provider（Qwen / Moonshot）

**背景：** Qwen、Moonshot 都提供 OpenAI 兼容的 `/v1/chat/completions` 接口。

**决策：** 一个 `OpenAICompatibleProvider` 类，通过 `base_url` + `env_key` 参数区分不同厂商，避免为每家写一个 provider。

**代价：** 无法使用各厂商的特有功能。当前阶段可接受。

### 决策 1b：DeepSeek 使用独立 provider

**背景：** DeepSeek R1（deepseek-reasoner）有独有的 `reasoning_content` 字段，在多轮 tool loop 中必须将其回传给 API，否则返回 400。通用 `OpenAICompatibleProvider` 无法处理这一协议细节。

**决策：** 新增 `DeepSeekProvider`，专门处理 reasoning_content 的提取和 round-trip。

**代价：** 多维护一个 provider 文件。

### 决策 2：工具调用循环上限 10 轮

**背景：** 防止 Agent 陷入无限循环消耗 token。

**决策：** `ToolUsingAgent` 硬编码最多 10 次工具调用迭代。

**代价：** 复杂任务可能被截断。后续可改为可配置参数。

### 决策 3：搜索工具用 DuckDuckGo

**背景：** 量化学习场景需要搜索能力，但不想强依赖付费 API。

**决策：** 使用 `duckduckgo-search` 库，无需 API key，开箱即用。

**代价：** 搜索质量和稳定性不如 Google/Bing API。

### 决策 4：raw_blocks 作为 provider 协议载体

**背景：** Anthropic 要求 tool_use 块保留在 assistant 消息的 content 数组中，tool_result 块放在后续 user 消息里；OpenAI 要求 tool_calls 字段在 assistant 消息上，tool 角色消息携带 tool_call_id。两种协议结构不同，无法用纯文本统一。

**决策：** `Message` 和 `LLMResponse` 均增加 `raw_blocks: list[Any]` 字段，携带 provider 原始结构化块。`ToolUsingAgent` 的 tool loop 保留并传递 raw_blocks，各 provider 在序列化时按自身协议处理。

**代价：** Message 对象携带了 provider 感知的数据，core 层与 provider 层有轻微耦合。

### 决策 5：LLMResponse.metadata 存放扩展数据

**背景：** DeepSeek R1 的 reasoning_content 需要在多轮 tool loop 中传递，但不属于标准 content 字段。

**决策：** `LLMResponse` 和 `Message` 均增加 `metadata: dict[str, Any]` 字段，`ToolUsingAgent` 在构建 assistant message 时将 `response.metadata` 一并传入，确保 reasoning_content 不在多轮循环中丢失。

**代价：** metadata 是弱类型 dict，provider 之间约定靠文档而非类型系统。
